Development

We are talking about S3 object oriented systems (as opposed to S4 or R6)

  1. Compared to S3/S4 a package built on R6 will seem odd from the user point of view
  2. If you are using an R package and need to use the @ symbol to reference items (slots) then it is coded in S4. Typically this is best saved for working in teams on a large package. Generally speaking S4 is more strict than S3 - so more similar to compiled languages (R is a intepreted language)
Resources

When your package depends on other packages it may be necessary to specify which version (e.g., R (>=4.05) in the Depends field of your DESCRIPTION file) and/or keep the versions up to date as well as Rtools.

Set min_version to TRUE to use the currently installed version.
usethis::use_package("ggplot2", type="Suggests", min_version=TRUE)

Workflow

  1. Create a dev folder and dev.R script
  2. usethis::use_build_ignore("dev")
    1. This creates a place to keep track of how you built the package without the errors/messages associated with running devtools::check()
  3. Create functions via usethis::use_r("function_name")
  4. Commit to git
  5. Document via devtools::document()
  6. Build a website using pkgdown - start with usethis::use_pkgdown() (once), then use pkgdown::build_site(examples=FALSE) to build a webpage
  7. commit to git
  8. push to github
  • Example dev code

Datasets

  1. They are publicly available and exist outside the NAMESPACE - they cannot be exported
    1. However you can (and I recommend) documenting them with a data.R file that is exported
    2. To bring data into an R package bring it into the Global Environment then usethis::use_data('data_set_name') - this will be stored in a data folder in the package stored in at the directory level

Using other packages

To use a package from CRAN
usethis::use_package("package_name")
Generally this should go to the “Imports:” section (default).

To use a package from github
usethis::use_dev_package("package_name", remote="BenWilliams-NOAA/package_name")

Remote Package Fix

Your DESCRIPTION file has to have the package in both the Imports: and Remotes: section

Imports: 
    package_name (>= 0.1.0),
Remotes:  
    BenWilliams-NOAA/package_name

Git/GitHub

Solo workflow

  • pretty straitforward regularly push/pull, use branches when you don’t want to blow things up.

Group workflow

  1. Gatekeeper - Team has limited or unknown git/GitHub skills
    One person(s) control whether changes are included in the package, and how they are integrated. All changes from the team are routed as a pull request, the gatekeeper is responsible for reviewing and inlcusion of all input (e.g., dealing w/merge conflicts)
  2. Free-for-all
  3. Equal - Team has good understanding of git/Github workflow

Be mindful of whom you give direct access to a repo (though you can always revert back if it blows up). One of the more challenging aspects to deal with, aside from git understanding, is different code styles - it is helpful to use one style within a package and be consistent throughout.

Git workflow
  • When working as a team only push the minimum commit necessary to update the code.
  • For example if a function is changed, only push the function commits, do not push up the associated documentation files.
  • This will reduce merge conflicts.
  • Use discrete commits and commit messages.
  • As a team decide where the cutoff is between issues and pull requests.
